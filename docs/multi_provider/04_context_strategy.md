# Per-Model-Family Context Strategy

**Status:** Design proposal · **Audience:** Stage 7 implementer + anyone touching agent context files.
**Scope:** Designs an extensible context-file system that supports many providers and models, replacing the current CLAUDE.md / AGENTS.md / GEMINI.md auto-sync pattern. Research-only: no existing files modified by this stage.

The migration in §5 keeps existing CLI tools (Claude Code, Codex, Aider, Cursor, Cline, Gemini CLI) working through the transition.

The CLI-tool family (CLAUDE.md, AGENTS.md, GEMINI.md, .aider.conventions.md, .clinerules, .cursorrules) is generated by `agent/sync_context.py`. The transform is almost a no-op — `strip_claude_specific()` only rewrites the H1, then prepends a per-tool header. Quoting `sync_context.py:56–68`:

```python
def strip_claude_specific(content):
    """Remove Claude-specific references and make content agent-generic."""
    result = re.sub(
        r"^# ImageJAI Agent — Claude Context",
        "# ImageJAI Agent Context",
        content,
    )
    return result
```

That's it. Five context files diverge from CLAUDE.md by exactly one heading line.

### What's in `agent/gemma4_31b/GEMMA_CLAUDE.md`

Completely different paradigm. Gemma has no shell or filesystem access — runs inside a Python tool-loop (`agent/ollama_agent/ollama_chat.py` + `agent/gemma4_31b/loop.py`). Tools are JSON, exposed by the wrapper. So GEMMA_CLAUDE.md describes:

- A structured tool table (lines 14–31): `run_macro`, `run_macro_async`, `run_script`, `probe_plugin`, `threshold_shootout`, `describe_image`, `capture_image`, `region_stats`, dialog tools, `run_shell`. NOT `python ij.py` invocations.
- Gemma quirks (lines 56–66, 170–183):
  - "**Vocabulary**. *Filter* = intensity operation… *Threshold* = binarisation… *Segmentation* = threshold + mask + Analyze Particles. '10 filters' never means '10 threshold methods'" — Gemma confused these in the friction logs.
  - "**threshold_shootout is the answer, not reconnaissance**" — Gemma over-called the shootout for non-thresholding tasks.
  - The whole *Common macro mistakes (lint will block)* section is built around Gemma's actual hallucination patterns.
- Infrastructure-specific text (lines 36–42, 73–87): triage banners, dismissed-dialog handling, post-timeout state.
- Reading tool references via `run_shell("type agent\\references\\…")` (line 122) — Gemma has no native filesystem read tool.

`agent/ollama_agent/CLAUDE.md` (developer note about the wrapper):
> **The model is Gemma 4 (4B local / 31B cloud). It is not Claude.** Single-purpose tools only. The docstring *is* the schema; `ollama.chat(tools=[fn, ...])` introspects.

So GEMMA_CLAUDE.md is a *family-tuned overlay*, not "CLAUDE.md minus 'Claude'".

About 90% of the content is universal. About 10% is genuinely model-family-specific. The current system puts 100% of the universal content in 7 files and pays the duplication tax 7×.

### Concrete failure modes today

1. The Groovy/Jython console error fact is missing from AGENTS.md / GEMINI.md / .aider.conventions.md. Codex/Aider/Gemini agents reflex-call `get_log` on script errors and find nothing.
2. Claude-isms in AGENTS.md / GEMINI.md / .aider.conventions.md tell Codex/Aider/Gemini to advertise `vision=true, token_budget=20000`. None of these tools actually do that.
3. GEMMA_CLAUDE.md duplicates the macro-reference pointer, recipe-book pointer, no-Enhance-Contrast rule, AI_Exports/ rule independently. If the Fiji rule changes, four files need manual updates.
4. There is no shared place to record a fact like "selectImage anchors prevent multi-image confusion" — discovered through Gemma friction, but every agent on a multi-image session benefits.

### Two distinct agent paradigms (key insight for §3)

Two **harnesses**, not one:

- **CLI-tool harness.** Claude Code, Codex CLI, Aider, Cursor, Cline, Gemini CLI. Has shell + filesystem tools (Bash, Read, Write, Edit). Controls Fiji by typing `python ij.py macro '…'`. CLAUDE.md / AGENTS.md / GEMINI.md target this.
- **Tool-loop harness.** The Ollama+Gemma wrapper (`agent/ollama_agent/ollama_chat.py`, `agent/gemma4_31b/loop.py`). The model emits structured tool calls; the wrapper executes. Tools: `run_macro`, `capture_image`, `threshold_shootout`. GEMMA_CLAUDE.md targets this. Multi-provider in `00` routes most new providers (OpenAI, Anthropic, Gemini native, Groq, Cerebras, OpenRouter, Mistral, …) through this harness.

Base content is the same across both; interface description (commands vs. tools) is not. **Harness is the first overlay axis. Family/model is the second.**

## Section 2 — Base context (`agent/contexts/base.md`, draft)

The base file contains every fact true regardless of provider, harness, or model. Single source of truth for Fiji house rules, workflow, references. Overlays add the harness-specific interface description and any family quirks.

Drafted in full below. Length target: ≤ 220 lines so a 4 k-context model can take it whole; harness overlay adds ≤ 80 more.

````markdown
# ImageJAI Agent — Base Context

You are an AI agent that helps a biologist analyse microscopy images
in ImageJ / Fiji. You read images, choose tools, write the macro,
inspect the result, and iterate until the measurement is sound.

This file is the universal context. The host harness will append a
section telling you exactly how to invoke the Fiji TCP server (either
via shell commands or via structured tool calls) and may add a few
notes about your specific model family. Everything in this file is
true for all agents.

---

## Your job

The biologist describes what they want in plain English ("count
GFP-positive cells in this stack", "is there colocalization between
channel 1 and 2", "render this neurone in 3D"). You translate that
into a Fiji workflow, run it, **look at the result**, and report
measurements with units. You do not guess biology from an image
description.

---

## Workflow

1. **Check state** — what images are open, what is selected, what
   tables already exist. Never assume an image is open.
2. **Check metadata** — is the image calibrated (μm vs px)? What
   are the channels, time points, z-slices? Bio-Formats metadata
   matters for measurement units.
3. **Probe unfamiliar plugins before using them.** The plugin
   probe opens the dialog, reads every parameter, and returns the
   real macro argument keys. Trust the probe over training-data
   memory — argument names change between Fiji versions.
4. **Check the recipe book** before building a workflow from
   scratch — `recipes/` holds known-good YAML pipelines.
5. **Plan in one short line.** Then act.
6. **Execute the macro.** Anchor any macro that touches the active
   image with `selectImage("<title>")` (or `selectWindow` for
   non-image frames like the ROI Manager). Title comes from the
   state check or the most recent open. Exception: macros that
   open a fresh image.
7. **Capture and look** — take a screenshot after every step that
   changes the image (threshold, segmentation, filter). For
   measurements use the numeric tools (histogram, region stats,
   line profile); the screenshot is for visual sanity, not
   numbers.
8. **Verify results** — read the measurements table. Sanity-check:
   counts in the expected order of magnitude, intensities not
   saturated, no NaNs.
9. **Check the log** for plugin warnings.
10. **Audit** — run the validator on the measurement table.
11. **Iterate.** If something looks wrong, fix the macro, retry.
    If the same error appears twice, stop and ask.

---

## House rules

- **Outputs go to `AI_Exports/`** next to the opened image. Never
  write elsewhere — the safety layer rejects it.
- **Never `Enhance Contrast` with `normalize=true` on data you
  will measure** — it permanently rewrites pixel values. Use
  `setMinAndMax(min, max)` for display-only adjustments.
- **Never close the ImageJ Log window during a session** — macros
  use it to surface warnings and progress.
- **Display segmentation results as a labelled object mask**, not
  as outlines or overlays — the user wants to see what was
  counted, not a hint at it.
- **Ask before silently picking** when two reasonable approaches
  exist (e.g., 3Dscript vs. 3D Viewer vs. 3D Project for a
  render). Present pros/cons.
- **Run the plugin scan once per session** so you know what is
  actually installed before suggesting a plugin.
- **Suggest plugin installs by name; never install
  programmatically.** The user opens *Help > Update… > Manage
  Update Sites > Apply > Restart*.
- **Stop on a repeated error.** Two identical macro errors, two
  identical probe rejections, two identical "No image" failures —
  do not send a third. Inspect state, ask the user.
- **`No image` always → check open windows first**, then retry.
- If a macro fails: read the error text. Fix the macro. Up to
  three attempts, then ask.
- **Fix the underlying bug.** If the TCP server, the probe cache,
  or a recipe is wrong, fix it — do not work around it silently.

---

## Vocabulary

These terms get confused. They are not interchangeable.

- **Filter** — an intensity operation on the image (Gaussian,
  Median, Mean, Unsharp, Bandpass). Output is still grayscale.
- **Threshold** — a binarisation method that picks an intensity
  cutoff (Otsu, Li, Triangle, Default). Output is still grayscale
  until you `Convert to Mask`.
- **Segmentation** — threshold + `Convert to Mask` + `Analyze
  Particles` (or StarDist, Cellpose, Labkit, …). Produces an
  object mask + counts.

"10 filters" never means "10 threshold methods". When the user
says one, do not substitute the other.

---

## Looking at images

You can see images. After any step that changes the image, take a
screenshot. The host harness will tell you the exact mechanism — in
some harnesses you call a screenshot tool and the image is
auto-attached to your next turn; in others you save the PNG to a
known path and read it back with your file-read tool. Either way:
capture, look, decide.

For numbers, use the dedicated stats tools (histogram, region
stats, line profile). The visual screenshot is lossy — never
measure from it.

---

## Error handling

If a macro fails the response carries `success: false` and an
`error` string. Common causes:

- "No image open" — open one first.
- "Not a binary image" — threshold and convert to mask first.
- "Selection required" — make an ROI first.
- "Command not found" — the exact name does not match an ImageJ
  menu item. Use the plugin scanner output to find the right
  spelling.
- "Wrong/unknown arguments" — probe the plugin first.

**Groovy / Jython script errors do NOT land in `IJ.getLog()`** —
they go to `System.err`, captured separately by the TCP server's
console buffer. If your script call returned a bare error and
the Fiji log is empty, fetch the console buffer (the harness
exposes this) before retrying — the stack trace there will tell
you what actually failed.

If a macro hung and the harness reports a timeout, do not
blind-retry. Read the post-timeout state (what was open, what was
logged), look for a `[triage] PLUGIN OUTPUT` banner naming
something like `Objects map of X` or `Summary of X` (that means
the plugin actually succeeded before the timeout fired), and
verify with the results table.

---

## Making toolbar buttons

You can build custom Fiji toolbar buttons ("tools") for the user —
one-click shortcuts for workflows, dialogs, or agent callbacks.
Write the tool macro, drop it in a toolset under
`<Fiji>/macros/toolsets/`, install at runtime with
`MacroInstaller`, and append to `StartupMacros.fiji.ijm` to
persist across restarts. Full syntax + worked examples are in
`references/fiji-toolbar-tools-reference.md`. Parameterised
template: `recipes/install_toolbar_tool.yaml`.

## Common macro mistakes

These are Fiji-language pitfalls, not model-specific. The lint
layer blocks the worst of them; you should avoid them up front:

- `Analyze Particles` on a non-binary image — threshold and
  `Convert to Mask` first.
- `Convert to Mask` without `setOption("BlackBackground", true)`
  earlier.
- `setAutoThreshold` on a stack without the `stack` keyword.
- Backslashes in macro paths — use forward slashes on Windows
  too.
- `Enhance Contrast normalize=true` before measuring (see house
  rules).
- `roiManager("Measure")` without `roiManager("reset")` first.
- `run("Measure")` without `run("Clear Results")` first.
- `run("Plugin Name…")` with no argument string — opens a dialog
  and hangs.
- `waitForUser()` — freezes the session. Never use it.
- `run("Analyze Particles…", "… add …")` — the keyword is
  `add_to_manager`, not `add`.
- `run(A, B, C)` with three string args — `run()` takes at most
  `(name, args)`. For saving use `saveAs(type, path)` directly.
- `print("Method: " + n)` to surface a number — use
  `setResult("Label", row, value)` + `updateResults()` so
  results come back as clean CSV.

## Multi-series files (`.lif`, `.czi`, `.nd2`)

These are containers that bundle many images. A Leica `.lif`
typically holds 50–200 series. Never blindly open every series —
a 78-series file can be 16 GB of pixels.

Two harness tools handle this: list the series (cheap, no pixel
read) and open a specific list of 0-indexed series. The harness
will tell you the exact tool names. Do not fall back to a
multi-series macro `run` — the macro importer silently opens
only series 1 when several are listed.

Forward slashes in paths on Windows. For per-channel names,
OME-XML, batch-iterate-close patterns, see
`references/bioformats-multiseries-reference.md`.

When the user is new (no `lab_profile.json`), suggest running the

testing, parameter tuning, parameter sweep) and writes
`lab_profile.json` + `learnings.md`. Subsequent agents inherit
those defaults.
````

## Section 3 — Overlay dimensions

Hypothesis: contexts vary along five dimensions — native vs wrapped tool calling, vision-capable vs text-only, context window size, tool-call reliability, family quirks. Validated below.

### 3.1 Harness — CLI-tool vs tool-loop *(KEEP — primary axis)*

The interface description (commands vs. structured tools) is the single biggest difference and is orthogonal to model identity. Codex CLI and Aider use the same harness even though powered by GPT-5 and Claude. Gemma 4 cloud and (planned) GPT-OSS-120B will share the tool-loop harness despite one being Google and one OpenAI. Harness goes first.

- `agent/contexts/harness/cli_shell.md` — names the `python ij.py …` commands, the `agent/.tmp/` capture path, the file-read instruction.
- `agent/contexts/harness/tool_loop.md` — names `run_macro`, `capture_image`, `threshold_shootout`, etc., explains screenshots are auto-attached.

Collapses six near-identical CLI files into one shared overlay.

### 3.2 Vision capability *(KEEP — material when off, no-op when on)*

Validated by code: `agent/ollama_agent/ollama_chat.py:113–115` defines `BASE_MODEL = "gemma4:e4b"` (4B local, no vision) alongside `CLOUD_MODEL = "gemma4:31b-cloud"` (vision). Llama 3.3 70B and Qwen 2.5 Coder are text-only.

Behavioural difference:
- **Vision present**: "after every step that changes the image, capture and look".
- **Vision absent**: skip the screenshot; rely on `histogram_summary`, `region_stats`, `quick_object_count`, `describe_image`. Capturing wastes a tool call.

Concrete `no_vision.md`:

> You cannot see images. Do not call `capture_image` — the screenshot would be sent to a model that ignores it. Instead, after every step that changes the image, call `histogram_summary` (intensity distribution), `region_stats` (mean/stddev on a rectangle), and `quick_object_count` (connected-component count at a fixed cutoff). The numbers replace the visual sanity check.

6-line overlay. Without it a text-only model burns calls on `capture_image` that do nothing.

### 3.3 Context window size *(KEEP — small-context overlay only)*

Models in scope: Gemma 4 4B (8k–32k effective), Phi-4 (16k), Mistral Small (32k); vs. Claude Opus 4.7 (200k–1M), Gemini 2.5 Pro (1M+), Llama 3.3 70B (128k), GPT-OSS-120B (128k). `_CLOUD_CTX_LIMITS` at `ollama_chat.py:124–126` pins gemma4:31b-cloud at 131,072.

For ≥100k context, base + harness + family fits trivially. For <32k, even a full base.md takes 5–10% of budget and a long session can blow the window.

`small_context.md` tells the agent to:
- Skip the references catalogue at the bottom of `base.md`. Read on demand only.
- Prefer one-shot terse macros over multi-step plans.
- Trim diagnostic chatter; keep tool calls.
- Drop the recipe-saving offer at end of session.

For ≥100k context, no overlay.

### 3.4 Tool-call reliability *(KEEP — material for weak callers)*

Three tiers, evidenced from `docs/ollama/improvement-loop/conversations/`:

- **Excellent** (Claude, GPT-5, Gemini 2.5 Pro, GPT-OSS-120B). Native tool calling, large training data on tool use, high reliability on complex schemas.
- **Good** (Llama 3.3 70B, Qwen 2.5 Coder 32B, DeepSeek V3, Mistral Large, GLM-4.6). Native tool calling but occasional schema slips — wrong arg name, missing required field, retry usually fixes.
- **Weak** (Gemma 4, Phi-4, small local models). The wrong-slot-arg pattern in `agent/ollama_agent/CLAUDE.md:11–13`: "A tool with two or more parameters will be called with wrong-slot arguments". Friction logs (`2026-04-19-filter-shootout-hallucinated-commands.md`, `2026-04-19-filter-loop-nresults-trap.md`) full of this.

Two overlays:

- `weak_tools.md` — "Tools take a single semantic argument. Read the docstring example values literally. If you are about to pass two parameters to a tool, you are calling the wrong tool — pick a more specific one. On any tool error, do not re-issue the same call with the same arguments."
- `good_tools.md` — minimal: "On a schema error, read the error string for the missing/extra field name and retry once. Do not retry blind."

Excellent-tier models need neither overlay.

The yaml stores `tool_call_reliability: high|medium|low` (per `02 §1` schema). Loader maps: `high` → no overlay, `medium` → `good_tools.md`, `low` → `weak_tools.md`.

### 3.5 Model-family quirks *(KEEP — tightly scoped)*

Each family has 2–6 specific quirks. Concrete observed:

- **Claude.** Verbosity. Tendency to explain plans at length when user wants result. The capability handshake `vision=true, output_format=markdown, token_budget=20000` (CLAUDE.md:7–25). Claude-only.
- **Gemma 4.** Single-arg tools. "Filter ≠ threshold ≠ segmentation" vocabulary. Triage banner reading. Threshold-shootout-as-final-answer.
- **Llama 3.x.** Over-explanation, especially on errors — narrates the error rather than retrying. (Anecdote pending verification once Stage 3 wires it up.)
- **Qwen Coder.** Strong on Python/Groovy, weak on macro syntax — sometimes writes Python idioms inside macros. One-liner: "ImageJ macro syntax is C-like, not Python — `for (i=0; i<n; i++)`, not `for i in range(n)`."
- **Gemini.** Tool-result formatting differs from Anthropic — Gemini wraps `function_response` in a structured object, sensitive to non-JSON content. Wrapper concern (Stage 3), not context. Empty overlay.
- **GPT-5 / GPT-OSS.** Fewest documented quirks. Likely empty overlay.
- **Phi-4 / DeepSeek / GLM.** Small-context families. Quirks expected, not yet observed — overlays start empty and accrue.

### 3.6 Dimensions explicitly NOT used

- **Streaming vs non-streaming.** Wrapper concern (`ollama_chat.py:910` notes `stream=False is required` because Ollama drops `tool_calls` in streaming mode). In wrapper, not prompt.
- **Reasoning vs non-reasoning models.** The `o3` / `claude-thinking` prompting tweaks matter at *router* level. Base "Plan in one short line. Then act." is correct for both.
- **Quantisation level.** Not actionable from a context file.
- **Provider routing details (LiteLLM vs native).** Not a model concern.

### 3.7 Final dimension list

```
harness × vision × context_size × tool_reliability × family
   2          2          2              3              ~10
```

Multiplicatively hundreds of permutations — but as overlays they compose, so file count is `2 + 2 + 1 + 2 + 10 ≈ 17 small files` (+ base). 4× reduction from today's 7 large files plus future expansion that would otherwise be one large file per model.

## Section 4 — Loading mechanism

### Recommendation: **Option A (additive overlays)**

**Option A.** Registry maps `(provider, model)` → list of overlays. Final context = `base.md` + each overlay's content concatenated. Each overlay is a small standalone file (10–80 lines). Registry is YAML.

**Option B.** Every model gets its own complete file in `agent/contexts/<provider>/<model>.md` opening with `@import base.md` and similar.

Option A wins:

1. **Defaults compose, exceptions stay small.** New model: 3-line registry entry (`harness: tool_loop`, `vision: false`, `family: qwen`). Option B requires a new file per model.
2. **Cross-cutting fixes propagate.** New universal Fiji rule? Edit `base.md` once. Option B requires editing every model file.
3. **The registry is the dropdown UI's surface.** Stage 5 reads the same yaml. One source of truth.
4. **Diff inspection.** A→B shows only differing overlays. Option B diffs two ~300-line files for a 20-line delta.
5. **No custom directive syntax.** `@import` would be a tiny home-grown templating language. Composition is just list concatenation — much harder to misuse.

Option B's "single readable file" property is recovered by a `--dump <model>` command that materialises the composed context to stdout.

### Directory layout

```
agent/contexts/
├── base.md
├── harness/
│   ├── cli_shell.md
│   └── tool_loop.md
├── capabilities/
│   ├── no_vision.md
│   └── small_context.md
├── reliability/
│   ├── good_tools.md
│   └── weak_tools.md
└── families/
    ├── anthropic.md  openai.md  gemini.md  llama.md  qwen.md  gemma.md
    └── phi.md  mistral.md  deepseek.md  glm.md  kimi.md  gpt_oss.md
```

### Registry: the unified `agent/providers/models.yaml`

The registry is the same `agent/providers/models.yaml` from `02 §1` — extended with context-loader fields. Both Java `ProviderRegistry` (UI side) and Python `loader.py` (system-prompt side) read it; each ignores fields it doesn't recognise. **One file, two consumers, no drift.**

The loader needs `harness`, `family`, `tool_call_reliability`, `vision_capable`, `context_size` from each entry; everything else (`display_name`, `description`, `tier`, `pinned`, etc.) is for the dropdown.

### Loader sketch (~30 lines)

```python
# agent/contexts/loader.py
from pathlib import Path
import yaml

CTX_DIR = Path(__file__).parent
REGISTRY = CTX_DIR.parents[0] / "providers" / "models.yaml"
# i.e. agent/providers/models.yaml — the unified registry from 02 §1


def load_context(model_id: str) -> str:
    """Compose the full agent context for `model_id` ("provider/model_id" form).

    Returns markdown. Raises KeyError if model is not registered."""
    cfg = yaml.safe_load(REGISTRY.read_text(encoding="utf-8"))
    provider, mid = model_id.split("/", 1)
    by_key = {(e["provider"], e["model_id"]): e for e in cfg["models"]}
    spec = by_key.get((provider, mid))
    if not spec:
        raise KeyError(f"unknown model id: {model_id}")

    parts = [(CTX_DIR / "base.md").read_text(encoding="utf-8")]
    parts.append((CTX_DIR / "harness" / f"{spec['harness']}.md").read_text(encoding="utf-8"))
    if not spec.get("vision_capable", True):
        parts.append((CTX_DIR / "capabilities" / "no_vision.md").read_text(encoding="utf-8"))
    if spec.get("context_size") == "small":
        parts.append((CTX_DIR / "capabilities" / "small_context.md").read_text(encoding="utf-8"))
    rel = spec.get("tool_call_reliability", "high")
    if rel == "medium":
        parts.append((CTX_DIR / "reliability" / "good_tools.md").read_text(encoding="utf-8"))
    elif rel == "low":
        parts.append((CTX_DIR / "reliability" / "weak_tools.md").read_text(encoding="utf-8"))
    family_path = CTX_DIR / "families" / f"{spec['family']}.md"
    if family_path.exists():
        parts.append(family_path.read_text(encoding="utf-8"))
    return "\n\n---\n\n".join(parts)


if __name__ == "__main__":
    import sys
    print(load_context(sys.argv[1]))
```

~32 lines including imports. `python -m agent.contexts.loader anthropic/claude-opus-4-7` materialises the full composed context to stdout.

Composition order: base → harness → capabilities → reliability → family. Earliest = most general; later overlays sharpen earlier text but never contradict (lint check in CI: every overlay declares what it overrides, if anything). Each overlay appended with `\n\n---\n\n` separator. No string interpolation, no template engine.

### What the wrapper does with the composed context

For tool-loop models: the composed string becomes the `system` message. For CLI-tool harnesses: written to disk on session start at the path the host CLI auto-reads (CLAUDE.md, AGENTS.md, GEMINI.md, etc. — see §5 migration). Wrapper handles routing; loader just produces text.

## Section 5 — Migration plan

**Hard constraint.** `agent/CLAUDE.md`, `agent/AGENTS.md`, `agent/GEMINI.md`, `agent/.aider.conventions.md`, `agent/.cursorrules`, `agent/.clinerules`, `agent/gemma4_31b/GEMMA_CLAUDE.md` must remain valid agent contexts at every commit. Live agents read these; deleting any breaks the user's current workflow.

**Strategy.** Build the new system alongside the old. `sync_context.py` is **rewritten** to delegate to `loader.load_context()`; the file path stays for backward compat with anyone running it manually, but the logic is replaced (per `07` Open Q3 RESOLVED).

(Step letters are local to this section — distinct from `07`'s Phase A–H.)

### Step 1 — Build (no live changes)

1. Create `agent/contexts/` with `base.md`, harness/capability/reliability/family overlays. Land `loader.py`. Add `python -m agent.contexts.loader --dump <model_id>`.
2. Add a snapshot test: `tests/test_contexts.py` runs `load_context()` for every model in the registry, asserts no missing-overlay errors, snapshots the output. CI fails on accidental drift.
3. Content-equivalence check: composed context for Claude Opus must contain every paragraph currently in `agent/CLAUDE.md` (modulo intentional drops like the broken `vision=true, token_budget=20000` lines that refer to a handshake that does not survive contact with the multi-provider system). A small Python script computes set-difference of bullet text and reports anything dropped that wasn't deliberate.
4. The existing `agent/CLAUDE.md` and siblings stay byte-identical. Live agents see no change.

### Step 2 — Cut over `sync_context.py`

`sync_context.py` already re-derives AGENTS.md / GEMINI.md / .aider.conventions.md / .cursorrules / .clinerules from CLAUDE.md. Change *what it derives from*, not the output paths.

1. Replace `read_claude_md()` with `loader.load_context("anthropic/claude-opus-4-7")` for the Claude path; `loader.load_context("openai/gpt-5-codex")` for AGENTS.md (Codex CLI); `loader.load_context("google/gemini-2-5-pro")` for GEMINI.md. Each output path corresponds to one model id.
2. Model id chosen for each output is the *primary user* of that file — Claude Code reads CLAUDE.md → Claude composition; Codex reads AGENTS.md → Codex composition; Gemini CLI reads GEMINI.md → Gemini composition. This *fixes* the §1 bug where AGENTS.md/GEMINI.md currently carry Claude-only handshake instructions.
3. Add a banner: `# Auto-generated by agent/contexts/loader.py — edit base.md / overlays / models.yaml`. Existing tools that grep for "Auto-generated from CLAUDE.md" still match because we leave that string adjacent.
4. Run once. Diff. Walk through each tool's regenerated context with copy/paste sanity check:
   - Codex CLI starts session against AGENTS.md — verify it does not see the Anthropic handshake noise it sees today.
   - Claude Code starts session against CLAUDE.md — verify the Groovy/console-error fact survived.
   - Gemma loop reads `GEMMA_CLAUDE.md` — Step 3.

After Step 2, the seven legacy CLI files are still there, byte-stable across reruns, but content is now derived from the overlay system. No live agent broken; six now see a more accurate context.

### Step 3 — Cut over the Gemma tool-loop

`agent/gemma4_31b/loop.py` injects a hand-built system prompt (or reads `GEMMA_CLAUDE.md` at startup — verify which by checking `loop.py`). Switch to `loader.load_context("ollama/gemma4:31b-cloud")`. Diff the output, accept deliberate changes (vocabulary section moves from Gemma file to base.md, but agent still sees it).

Keep `GEMMA_CLAUDE.md` regenerated by `sync_context.py` so anyone reading manually sees live context. Mark with banner.

### Step 4 — New providers wire the loader directly

When Stage 7 lands `agent/providers/<provider>/wrapper.py`, every wrapper calls `loader.load_context(model_id)` directly — no intermediate files for new tool-loop providers. Only legacy CLI tools need regenerated files on disk.

### Step 5 — Eventual cleanup (months later)

After every CLI tool the user actually uses has been verified, the legacy files are regenerated artefacts only. Checked in (because the user clones and runs Claude Code immediately) but auto-derived. No further action unless we drop a CLI tool entirely; in that case, delete the corresponding output path from `sync_context.py` and the file.

### Risks

1. **Loss of fidelity.** A hand-edited line in current CLAUDE.md gets dropped. Mitigation: Step 1 content-equivalence check; explicit allow-list of intentional drops; one full review of the diff before flipping.
2. **A live CLI tool reads a file we missed.** Mitigation: project-wide grep on Step 1 start (`agent/CLAUDE.md`, alternates `CONVENTIONS.md`, `INSTRUCTIONS.md`, hidden dotfiles).
3. **Registry drift.** Adding a new model to the dropdown without a registry entry causes `KeyError` at session start. Mitigation: dropdown loads its model list from the registry, so an unregistered model simply doesn't appear.
4. **Family quirks regress.** A family overlay hand-tuned for Gemma might not survive a refactor. Mitigation: Step 1 snapshot test locks composed output per model. Quirk changes show as snapshot diff and are reviewed.

## Summary

- **§1.** Today's seven CLI files are 90% duplicate; auto-generated from CLAUDE.md by a one-line transform. GEMMA_CLAUDE.md is a separate paradigm (tool-loop, not CLI). Two failure modes: Claude-only handshake leaks into AGENTS.md/GEMINI.md/Aider; Groovy/console-error fact only made it into CLAUDE.md.
- **§2.** A 215-line `base.md` covers everything universal. Lifts "selectImage anchor" and lint-block list from GEMMA (Fiji rules, not Gemma rules) and adds MEMORY.md house rules.
- **§3.** Five overlay dimensions actually matter: harness (cli_shell vs tool_loop), vision (yes/no), context size (default vs small), reliability (high/medium/low → no overlay/good_tools/weak_tools), family (~10 entries). Streaming, reasoning-style, quantisation do not belong in the context.
- **§4.** Additive overlays (Option A) win. ~32-line Python loader composes `base + harness + capabilities + reliability + family` keyed by `agent/providers/models.yaml` (the unified registry — no separate context yaml).
- **§5.** Five-step migration (steps 1–5, distinct from `07`'s phase letters). `sync_context.py` is rewritten to delegate to the loader. Live CLI tools never break.
- **§6.** Drafted descriptions for 17 models, mapping technical traits to lay phrases.
